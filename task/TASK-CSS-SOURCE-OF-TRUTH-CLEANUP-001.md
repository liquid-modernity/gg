## 1. Purpose

The current repo has too many files that look editable but may not be part of the actual build/bundle/deploy graph.

This creates a dangerous failure mode:

```txt
User or CODEX edits a file that looks like a source file
↓
build still passes
↓
live site does not change
↓
the repo now contains misleading zombie source
```

This task exists to remove that ambiguity.

The goal is to classify, document, wire, or delete stale CSS/JS/module/artifact files so that the repo has a clear source-of-truth model:

```txt
src/        = canonical human-editable source
assets/     = route/runtime public assets, copied or built from source where applicable
__gg/       = generated/staged public GG app assets used by the Blogger/Cloudflare runtime
dist/       = generated build artifacts
.cloudflare-build/ = temporary Cloudflare prepared deployment output
```

The task must not become a visual redesign, runtime rewrite, or broad refactor.

## 3. Hard Constraints

### 3.1 Do not break stable contracts

Do not change:

- Blogger route truth
- Cloudflare Worker route truth
- `/landing = Home`
- `/ = Blog`
- `/store = Store`
- breadcrumb truth: `Home(/landing) -> Blog(/) -> current`
- Blogger-native comments plumbing
- Store runtime
- Discovery runtime
- Preview runtime
- Shell runtime
- Theme runtime
- Blog1 rendering behavior
- Worker healthy-route non-authoring behavior

### 3.2 Preserve Blog1-safe schema

Do **not** reintroduce:

```txt
data:schemaPosts
dynamic root ItemList
filtered data:posts schema loops inside root Blog1
```

Root `/` schema may remain Blog + BreadcrumbList only unless a future task proves dynamic ItemList is Blogger-safe on live rendering.

### 3.3 No fake cleanup

No deletion is allowed without proof.

For every removed file, block, selector, script, or artifact, document:

```txt
- exact removed path/block
- where usage was searched
- whether package.json references it
- whether tools/*.mjs references it
- whether qa/*.mjs references it
- whether index.xml / landing.html / store.html references it
- whether Cloudflare prepare/deploy references it
- whether Store build references it
- whether generated artifacts include it
- why it is unused/stale
- what guard protects against regression
```

### 3.4 No override-only fixes

Do not solve ambiguity by adding another CSS/JS override layer.

If a module is real, wire it into the build/bundle graph.
If a module is stale, delete it with proof.
If a file is generated, document it as generated and do not edit it manually.

## 4. Required Folder Classification

Create or update documentation so a non-technical owner can understand the folder roles.

At minimum, document these:

### `src/`

Canonical human-editable source.

Expected examples:

```txt
src/css/gg-app.source.css
src/css/gg-critical.source.css
src/css/components/*
src/css/modules/*
src/js/gg-app.source.js
src/store/*
src/registry/*
```

Rule:

```txt
Edit source here first.
Do not edit generated output as the primary fix.
```

### `assets/`

Public route/runtime assets.

Expected role:

```txt
assets/store/*
```

These may be copied/built from `src/store/*`.

Rule:

```txt
Do not edit assets/store output manually if src/store is the source-of-truth.
```

### `__gg/`

Generated/staged public GG app assets used by the runtime.

Expected examples:

```txt
__gg/assets/css/gg-app.dev.css
__gg/assets/css/gg-app.min.css
__gg/assets/js/gg-app.dev.js
__gg/assets/js/gg-app.min.js
```

Rule:

```txt
Do not edit __gg/assets manually.
They must match source/bundle output.
```

### `dist/`

Generated build/publish artifacts.

Expected examples:

```txt
dist/blogger-template.publish.xml
dist/assets/css/*
dist/assets/js/*
```

Rule:

```txt
Generated by template/build tools.
Do not edit manually.
```

### `.cloudflare-build/`

Temporary Cloudflare prepared deploy output.

Rule:

```txt
Generated by tools/cloudflare-prepare.mjs.
Never edit manually.
Usually not committed unless the repo intentionally does so.
```

## 14. Practical Explanation For Repo Owner

Use this mental model:

```txt
src/     = tempat edit manusia
assets/  = asset publik route tertentu, sering hasil copy/build dari src
__gg/    = asset publik aplikasi GG yang sudah disiapkan untuk runtime
dist/    = hasil build/publish
.cloudflare-build/ = paket sementara untuk deploy Cloudflare
```

If a file is not read by the build, not loaded by the site, not referenced by QA, and not documented as advisory/manual, it should not stay in the repo.

A repo that contains fake source files is worse than a repo with fewer files.
