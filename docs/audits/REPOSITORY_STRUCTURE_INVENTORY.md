## Root-Level File Analysis

The root is intentionally not a textbook source-only root because Blogger, Cloudflare, PWA, and public static routes depend on stable root filenames. `index.xml`, `landing.html`, `store.html`, `worker.js`, `_headers`, `robots.txt`, `manifest.webmanifest`, `sw.js`, `offline.html`, and flag JSON files should remain at root unless a future architecture task updates every owning tool, guard, workflow, and public contract.

The root is less professional where historical reports, task archives, copy JSON aliases, and ZIP archives sit beside runtime files. Some root reports are still guard-referenced by exact path, so moving them is not a low-risk tidy action. Root ZIP/archive files are candidates for a separate cleanup or archive-retention task, not for this planning task.

## Naming and Clutter Observations

- Files or directories with spaces exist in `src/knowledge base`, `assets/knowledge base`, and `task/task-004-audit local npm ci esbuild compatibility.md`.
- Copy registry names appear in both root JSON aliases and `registry/copy/*`; guards currently expect this relationship.
- Historical backup/archive markers exist in `task/index-backup.xml`, `registry/original/*`, `docs/archive/template-deprecated/*`, and root/task ZIP files.
- Generated naming such as `dist/index.final.xml`, `dist/index.final.production.xml`, and Store generated route artifacts is build-owned and should not be renamed casually.
- Local ignored clutter can reappear during macOS Finder use, including `.DS_Store`.
